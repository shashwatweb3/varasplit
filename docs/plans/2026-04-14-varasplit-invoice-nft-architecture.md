# VaraSplit Invoice NFT Architecture

## Summary

The dApp remains a single Sails program and one `VaraSplit` service. Settlement now mints a stored invoice NFT record and returns a slim `GroupSettlement` reply. The frontend queries the minted NFT by token id to display the invoice.

## Program And Service Boundaries

- Program constructor: `create() -> Self`
- Service: `vara_split`
- Public routes are Sails exports only.
- No raw payload route or hand-built SCALE payload is introduced.

## State Ownership

`Program` owns `RefCell<VaraSplitState>`.

`VaraSplitState` owns:

- group id allocation
- token id allocation
- group ledgers
- invoice NFT records

Groups store balances as `Vec<MemberBalance>` to match the required ABI. Helper functions search balances by member using iterators and `get`-style option handling.

## Message Flow

- `CreateGroup`: validates creator as first member, de-duplicates, creates zero balances, stores group, emits `GroupCreated`.
- `AddExpense`: validates caller/payer/amount/description, splits debt across balances with checked math, stores expense, emits `ExpenseAdded`.
- `ConfirmPayment`: validates caller, members, direction, and amount, then reduces debt with checked balance updates and emits `PaymentConfirmed`.
- `SettleGroup`: validates caller and one-time invoice mint, computes transfers, allocates token id, stores `InvoiceNFT`, emits `InvoiceNFTMinted`, resets balances, emits `GroupSettled`, returns `GroupSettlement`.
- `GetInvoiceNFT`: queries invoice NFT by token id.

## Routing And Public Interface

Existing stable routes:

- `CreateGroup`
- `AddExpense`
- `ConfirmPayment`
- `SettleGroup`
- `GetGroup`
- `GetBalances`
- `GetSettlementPlan`

New route:

- `GetInvoiceNFT`

Removed route:

- `GetInvoice`

Method signature or reply shape changes:

- `GroupSettlement` no longer contains invoice data.
- Invoice data is exposed through `InvoiceNFT`.
- `Group` and `GroupView` expose balances as `Vec<MemberBalance>`.

## Event Contract

Existing events stay present:

- `GroupCreated`
- `ExpenseAdded`
- `PaymentConfirmed`
- `GroupSettled`

New event:

- `InvoiceNFTMinted { token_id, owner, group_id }`

Event versioning:

- Not required for this unreleased MVP workspace, but enum order for `VaraSplitError` is frozen.

## Generated Client Or IDL Impact

- IDL must be regenerated after the release build.
- Rust client must be regenerated by the repo build path.
- Frontend IDL must be replaced with the new IDL and must not reuse the old `GetInvoice` ABI.
- Frontend must pass group ids and token ids as `BigInt` at the Sails boundary.
- Frontend must pass account ids as `Uint8Array` from `decodeAddress`.

## Contract Version And Status Surface

- No explicit version route.
- No read-only lifecycle state.
- No deployed cutover concerns in this local MVP task.

## Off-Chain Components

- Wallet connect remains client-only using Polkadot extension.
- Local storage may remember the selected address for auto reconnect.
- Group settlement UI should show the settlement reply immediately and fetch/display the invoice NFT when token id is known.
- Since `SettleGroup` returns no token id, the frontend can query token id `0` for the first settlement in an MVP flow or use event decoding later. Current production-facing adapter exposes `getInvoiceNFT(tokenId)`.

## Failure And Recovery Paths

- Contract validation errors return `VaraSplitError` through Sails `result`.
- Frontend must branch on Sails result errors before normalizing replies.
- If IDL or generated client differs from frontend assumptions, the IDL is source of truth.

## Open Questions

- Future work may add an explicit latest-invoice-by-group query or include `token_id` in `GroupSettlement`, but that would change the strict requested signatures and is not included here.
